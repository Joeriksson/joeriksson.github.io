Handling Periodic Tasks in Django with Celery

Intro

In your Django app there might be a need to run certain tasks periodically and automatically in the background. Example of such tasks might be:

  • Cleaning up cache
  • Sending e-mail reports/notifications
  • Running nightly maintenance jobs

There is no built-in functionality in Django to run scheduled tasks. But if we integrate for example Celery and Redis with our app, this can be accomplished. In the following article, we will show you how to set up Django, Celery, and Redis in a Docker container, and use a custom Django-Admin command to run periodic tasks.

Dependencies: 1. Django v3.0.5 2. Docker v19.03.8 3. Python v3.8.0 4. Celery v4.4.1 5. Redis v5.0.7

Objectives

By the end of this tutorial, you should be able to:

  1. Containerize Django, Celery, and Redis with Docker
  2. Integrate Celery into a Django app and create tasks
  3. Write a custom Django-Admin command
  4. Schedule the command to run periodically

Getting Started

Although you will in this article learn how to setup the containers step-by-step, for details on how the workflow between Django, Celery, and Redis works, have a look at Asynchronous Tasks with Django and Celery | TestDriven.io

Dockerize Django - Celery - Redis

Docker

If you haven’t already, install Docker and add a Dockerfile to the projects root directory.

# Pull base image
FROM python:3.8-slim

# Set env vars
ENV PYTHONDONTWRITEBYTECODE 1
ENV PYTHONUNBUFFERED 1

# Set working dir
WORKDIR /code

# Install dependencies
COPY requirements.txt /code/
RUN python -m pip install --upgrade pip
RUN python -m pip install -r requirements.txt

# Copy project
COPY . /code/

Here we are using an official Docker image for Python 3.8. Then we set two environment variables:

  1. PYTHONDONTWRITEBYTECODE which prevents Python from writing .pyc files to disc.
  2. PYTHONUNBUFFERED which forces the stdout and stderr streams to be unbuffered.

Then we copy the requirements.txt to the working directory, upgrade pip, and install the requirements and copy over the whole Django project to the container.

Django

To get our Django project to run in the Docker container we will add a docker-compose.yml file to the same directory as the Dockerfile.

version: '3.7'

services:
  web:
    env_file:
      - .env.dev
    build: .
    command: python /code/manage.py runserver 0.0.0.0:8000
    volumes:
      - .:/code
    ports:
      - 8000:8000

Next create a .env.dev file in the project root with the following variables:

DEBUG=1
SECRET_KEY=mysecretkey
DJANGO_ALLOWED_HOSTS=localhost 127.0.0.1 [::1]

If you cloned the repo for this article, the settings.py is already setup to read these environment variables. For example:

SECRET_KEY = os.environ.get(‘SECRET_KEY’)

Postgres

To also run our Postgres database in the container let's add a new service (called db) into the docker-compose.yml file.

version: '3.7'

services:
  web:
    env_file:
      - .env
    build: .
    command: python /code/manage.py runserver 0.0.0.0:8000
    volumes:
      - .:/code
    ports:
      - 8000:8000
# NEW
    depends_on:
      - db
  db:
    image: postgres:12
    volumes:
      - postgres_data:/var/lib/postgresql/data/
    ports:
      - 5432:5432
      environment:
        - POSTGRES_USER=postgres
        - POSTGRES_PASSWORD=postgres
        - POSTGRES_DB=postgres
volumes:
  postgres_data:

For the data in the database to persist after the container is shut down, we bind a volume postgres_data to the ‘/var/lib/postgresql/data/‘ directory in the container.

Build the new image and spin up the containers:

$ docker-compose up -d —-build

Run the migrations:

$ docker-compose exec web python manage.py migrate —noinput

Check that the volume was created by running:

$ docker volume inspect periodic_tasks_django_postgres_data

Celery and Redis

Add the following to the end of the docker-compose.yml file:

redis:
    image: redis:alpine

celery:
    env_file: .env
    build: .
    command: celery -A core worker -l info
    volumes:
      - .:/code
    depends_on:
      - db
      - redis
celery-beat:
    env_file: .env
    build: .
    command: celery -A core beat -l info
    volumes:
      - .:/code
    depends_on:
      - db
      - redis

We also need to update the web service depends_on section:

web:
    env_file:
      - .env
    build: .
    command: python /code/manage.py runserver 0.0.0.0:8000 --settings=core.settings.development
    volumes:
      - .:/code
    ports:
      - 8000:8000
    depends_on:
      - db
      - redis # NEW

Celery configuration

Before building the new containers we need to configure Celery in our Django app.

In the core directory, create a celery.py file and add the following code:

import os
from celery import Celery

os.environ.setdefault('DJANGO_SETTINGS_MODULE', 'core.settings')

app = Celery('core')
app.config_from_object('django.conf:settings', namespace='CELERY')
app.autodiscover_tasks()

Add the following code to core/__init__.py:

from .celery import app as celery_app

__all__ = ('celery_app',)

Build the new containers to see that everything works.

$ docker-compose up -d -—build

Take a look at the logs for each service to see that they are ready and without errors.

$ docker-compose logs 'web' $ docker-compose logs 'db' $ docker-compose logs 'celery' $ docker-compose logs 'celery-beat' $ docker-compose logs 'redis'

If all went well, we now have 5 containers with different services, and are ready to create a sample task to see that it works as it should.

Create a task

Create a new file core/tasks.py

from celery import shared_task


@shared_task
def sample_task():
    print('The sample task was run')

Schedule the task

At the end of settings.py, add the following code that will schedule sample_task to run once per minute.

from celery.schedules import crontab 
import core.tasks

CELERY_BEAT_SCHEDULE = {
'sample_task': {
    'task': 'core.tasks.sample_task',
    'schedule': crontab(minute='*/1'),
  },
}

Restart the container to be sure the new settings.py is read.

$ docker-compose down && docker-compose up -d

Take a look at the celery log in the container.

$ docker-compose logs 'celery'

Every minute you should see a row in the log that ends with ‘A scheduled task just ran’.

Create a custom Django-admin command

What is a Django-admin command? In this article we already used a django-admin commands: * python manage.py migrate

But Django also gives us the option to create our own custom commands. From the Django documentation

Custom management commands are especially useful for running standalone scripts or for scripts that are periodically executed from the UNIX crontab or from Windows scheduled tasks control panel.

To create our own custom command, all we have to do is to create a .py file in a management/commands directory.

Create a new file <appname>/management/commands/my_custom_command.py. The minimal code needed for a command is:

from django.core.management.base import BaseCommand, CommandError


class Command(BaseCommand):
    help = 'A description of the command'

    def handle(self, *args, **options):
        pass

The BaseCommand has a couple of methods that can be overridden. But the only one that is required is the handle method. When we run our custom command this is the code that is going to run.

We can add a print command to our custom command, so we can test that the command works. Instead of Pythons print statement, we will use stdout.write recommended in the Django documentation.

When you are using management commands and wish to provide console output, you should write to self.stdout and self.stderr, instead of printing to stdout and stderr directly. By using these proxies, it becomes much easier to test your custom command. Note also that you don’t need to end messages with a newline character, it will be added automatically, unless you specify the ending parameter:

Add a self.stdout.write command to our code.

from django.core.management.base import BaseCommand, CommandError


class Command(BaseCommand):
    help = 'A description of the command'

    def handle(self, *args, **options):
        self.stdout.write('My sample command just ran') # NEW

From the command line run:

$ docker-compose exec web python manage.py my_sample_command

You should get an output of 'My sample command just ran'.
